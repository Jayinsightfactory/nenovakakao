# work_bridge v2 재설계 기획서 — 워크→카톡 본문 중계

> 작성 2026-06-03. 현재 v1(미리보기 방식)의 근본 결함을 고쳐, "워크에 새로
> 생긴 메시지 전체를 카톡 원본방으로" 보내는 올바른 구조로 재설계.

## 1. 현재 v1 의 문제 (사용자 지적)

현재 `core/work_bridge.py` 는 **룸목록 preview(미리보기 한 줄)** 를 읽어 카톡으로 보냄.

- ❌ **미리보기 한 줄만** 감 — 전체 메시지 아님, 잘림
- ❌ preview 가 **카톡→워크로 미러된 내용**일 수 있음 → 카톡에 이미 있는 걸 되쏨(에코)
- ❌ 누가 보냈는지 / 카톡 것인지 워크 것인지 **구분 불가**
- 실제 사고: `"[심동욱] (오후 3:54) 22-2차 멜로디"`(카톡→워크 미러본)를 카톡으로 되쏨,
  `"사진 2장을 보냈습니다."`(미리보기 텍스트) 전송 등

## 1.5 운영 전제 (2026-06-03 확정 — 중요)

- **이 PC 는 모니터링 전용**. 사람이 이 PC 의 워크에 직접 답장 치는 일은 **없음**.
- 따라서 워크방의 새 메시지는 **항상 "남(관리자/팀원이 폰·타 기기)이 워크에 보낸 것"**
  → unread 증가로 확실히 탐지됨. ("내가 친 답장" 케이스 자체가 없음)
- **답장버튼 대안은 이 PC 에 부적합** → work_bridge v2 가 유일한 정답.
- 남이 워크에 보내는 일이 **자주 있음** → 구현 우선순위 높음.

## 2. 원하는 구조 (확정)

```
워크방에 사람이 직접 친 "새 메시지 전체 내용" → 카톡 원본방으로 전송.
단, 카톡→워크로 이미 미러된 내용/봇 메시지는 제외(카톡에 이미 있으므로).
```

핵심 원칙:
- **미리보기 아님 = 대화창 본문 전체를 읽는다**
- **카톡에 없는 것만** = 미러본/봇 메시지 제외
- **새 것만** = 직전에 처리한 메시지 이후의 신규분

## 3. 2단계 아키텍처

전체 방을 매번 열 수 없으므로(수십 개, 느림+크레딧), 가벼운 탐지 → 정밀 읽기.

### 1단계 — 새 메시지 있는 방 탐지 (가벼움)
- 룸목록 캡처(TOPMOST 안정화 적용) → `detect_unread_badge_rows` 로 **unread 뱃지 증가** 방 식별
- LLM 불필요(픽셀 스캔) 또는 1회 Vision 으로 방이름+unread 만
- 출력: "새 메시지 있는 워크방 목록" (보통 0~3개)

### 2단계 — 그 방만 열어 본문 읽기 (정밀)
- 후보 방을 **하나씩 열어**(워크 앱에서 클릭) → 대화창 캡처 → `extract_messages`(Opus)
  로 본문 메시지 리스트 추출 (발신자/시각/내용/has_image)
- 각 메시지를 필터:
  1. **봇 시스템 메시지** (`_is_bot_system_preview`) → 제외
  2. **카톡→워크 미러본** (`_looks_like_mirror_header`, "[발신자] [시각] …") → 제외
     (이게 "카톡에 이미 있는 것" 핵심 구분자)
  3. **이미 중계한 메시지** (sent ledger 해시) → 제외
  4. 나머지(= 사람이 워크에 직접 친 새 메시지) → ✅ 카톡 원본방으로 **전체 내용** 전송
- 전송 후 그 메시지 해시를 ledger 에 기록(재중계 방지)

## 4. 새 메시지 판별 — "워크에서 친 것" vs "카톡 미러본"

워크 미러방 한 대화창에는 두 종류가 섞임:
- **카톡 미러본**: 모니터가 `[발신자] [시각] 내용` 형식으로 넣은 것 → **카톡에 이미 있음** → 제외
- **워크 직접 입력**: 사람이 워크에서 친 raw 텍스트(형식 없음) → **카톡에 없음** → 전송 대상

판별: `_looks_like_mirror_header` 정규식(`^\[…\] \[시각\]`)으로 미러본 제외 →
남은 게 워크 네이티브 메시지. (이미 v1 에 구현된 필터 재사용)

## 5. 트리거·주기·크레딧

- 데몬 주기: 30~60초(조정). 1단계는 매 주기, 2단계는 후보 있을 때만.
- 크레딧: 1단계 픽셀 스캔(무료) + 2단계 후보 방당 Opus 1회. 후보 0이면 호출 0.
  → v1(매 주기 전체 Vision)보다 크레딧 절약.
- 모니터 경합: 2단계가 워크앱+카톡앱 둘 다 조작 → `_klock` 우선요청 +
  카톡 메인 전면화(v1 수정 반영) + 방열기 10s.

## 6. 무한 에코 방지 (다중 안전망)

1. `_looks_like_mirror_header` — 카톡 미러본 제외
2. `_is_bot_system_preview` — 봇 메시지 제외
3. sent ledger(`work_sent_recent` + `sent_ledger`) — 이미 보낸 것 제외
4. 카톡으로 전송 시 그 내용이 다시 카톡→워크 미러될 때, 위 1·3 이 재차단

## 7. 한계·리스크 (정직)

- **2단계가 워크방을 클릭해 열어야** 함 → 워크 앱 화면 자동화(좌표/포커스).
  카톡 서랍만큼 까다로움. 라이브 튜닝 필요.
- 워크방 많고 새 메시지 잦으면 2단계 Opus 호출 누적 → 크레딧·속도 부담.
- "내 PC 에서 내가 친 메시지"는 워크 unread 안 늘어 1단계가 못 잡을 수 있음
  (남이 보낸 건 unread↑로 잡힘). 내가 친 답장은 답장버튼이 적합.
- 모니터와 동시 운영 시 화면 경합 — 시분할 또는 우선권 조정 필요.

## 8. 구현 단계 (제안)

- P1: 1단계(unread 탐지) — 이미 부품 있음(`detect_unread_badge_rows`)
- P2: 워크방 열기 자동화(`open_work_room(name)`) — 신규, 라이브 튜닝
- P3: 2단계 통합(열기→본문 추출→필터→전송) + ledger
- P4: 모니터 경합/크레딧 가드 + E2E 검증

## 9. 대안 (참고)

- **답장버튼**(현행 가능): 사람이 워크에서 카톡 답장 시 가장 안정·크레딧 무관.
  "내가 치는 답장"엔 오히려 적합. work_bridge v2 는 "남이 워크에 보낸 것 중계"용.
- 둘 병행 가능하나 답장버튼 부착 시 preview 오염 → v2 는 본문 읽기라 preview
  무관, 병행 가능(단 답장버튼 메시지는 _is_bot_system_preview 로 제외됨).
